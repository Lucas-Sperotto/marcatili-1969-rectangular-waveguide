# 00.6 Revisao da Camada IO

Data: 2026-04-16.

Escopo revisado:

- `schema_json`, `text_io`
- `single_guide_io`, `coupler_io`
- `fig6_io`, `fig7_io`, `fig8_io`, `fig10_io`, `fig11_io`, `table1_io`

## Resumo

A camada IO esta bem estruturada e consistente com a arquitetura do projeto: parse de entrada, serializacao de saida e default de caminhos de artefatos por executavel.

Pontos fortes:

- limites claros entre `io` e `physics`;
- saidas rastreaveis em `JSON/CSV`;
- parser leve com suporte a chaves em caminho (`a.b.c`);
- defaults de arquivo de saida previsiveis.

## Achado corrigido nesta etapa

### F6-IO-001 - Campo `model` inconsistente em `fig6_io`

- Problema: `BuildFigure6JsonReport` escrevia sempre `"model": "closed_form_approximation"`, mesmo com sweep misto (`exact` + `closed_form`).
- Correcao aplicada: agora `model` reflete o escopo real:
  - campo renomeado para `app_model` com valor estavel `figure6_sweep`;
  - a lista real de solvers continua no campo `solver_models`.

### JSON-IO-002 - Estruturas compactas migrando para objetos

- Problema: listas como `variant_id:n2:n3:n4:n5` e `row_id:...` eram praticas, mas fragilizam manutencao.
- Correcao aplicada:
  - `fig6_io` e `table1_io` agora aceitam arrays de objetos JSON;
  - formato legado de strings compactas permanece suportado por compatibilidade.
  - `reproduce_table1.json` foi migrado para objetos.

### TBL-IO-004 - Hipotese da Tabela I explicitada no schema

- Problema: a interpretacao dos valores tabulados era sensivel e ficava diluida em texto.
- Correcao aplicada:
  - adicionado campo explicito `table_entry_interpretation`;
  - valor atual validado no parser/solver: `a_times_n1_over_lambda`;
  - relatorios `JSON/CSV` da Tabela I agora carregam essa informacao.

### CSV-IO-003 - Escaping robusto de texto em CSV

- Problema: campos de texto com virgula/aspas/quebra de linha poderiam invalidar CSV.
- Correcao aplicada:
  - adicionado `EscapeCsv` em `text_io`;
  - relatorios CSV dos apps passaram a usar escaping explicito para campos textuais.

## Riscos/limitacoes ainda abertas

1. `schema_json` e intencionalmente minimalista (escopo controlado do projeto), nao um parser JSON geral.
2. `RequireStringArrayValues` trata array vazio como ausente; hoje isso e aceitavel para os casos de reproducao, mas vale manter como convencao explicita.
3. Persistem ambiguidades cientificas de OCR (especialmente Fig. 8 e Fig. 10), que nao sao falhas de IO, mas afetam a interpretacao dos campos de entrada.

## Observacao cientifica (Fig. 10)

A implementacao segue uma hipotese documentada no repositorio: usar Eq. (34) com Eq. (6)/(12) para `k_x`, mesmo com ambiguidade de rotulo modal no scan.

Estado: **assunto aberto, tratado explicitamente**, nao erro oculto.
